# Ajouter les tests unitaires à son workflow d'intégration continue
- Ajoutez un fichier test_counter.py (à la racine):
```py
from counter import add_one_to_count, remove_one_to_count

def test_add():
    assert add_one_to_count(0) == 1

def test_remove():
    assert remove_one_to_count(1) == 0

def test_remove_no_change():
    assert remove_one_to_count(0) == 0

```
- Ajouter la dépendances dans le requirements.txt existant (ou vous voulez, exemple à la fin du fichier)
```
pytest
```
- Buildez les dépendances de votre projet (depuis le terminal, au même niveau que le fichier requirements.txt) dans un terminal :
```sh
pip install -r requirements.txt
```

## Démarrer vos tests en local
- En ligne de commande (toujours à la racine) :
```pytest``` ou ```python -m pytest```
- Normalement vos tests doivent passer

## Modification pour bandit qui detecte des erreurs mainteant : 
- En ligne de commande (toujours à la racine) :
```bandit -r . -x test_counter.py``` ou ```python -m bandit -r . -x test_counter.py```
- On doit exclure les fichiers de tests, car "assert" est pas considéré comme du code "safe", car c'est du code qui disparaît à la compilation et avec notre architecture de dossier simplifiée, bandit ne se rend pas compte que c'est un fichier de tests.
- Normalement, avec un code aussi basique, vous n'aurez aucune remontée d'erreurs

## Adapter votre CI (workflow)
- Ajouter la commande de lancement des tests unitaires avec pytest (En général, c'est réalisé juste après d'installation des dépendances)
- Pensez également à adapter la commande bandit !
- Là, on est une CI un peu plus complète, mais on va encore la compléter plus tard
